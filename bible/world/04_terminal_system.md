# 04 — 终端系统详解（Terminal System v2.0）

---

> **核心变更说明**：本版本对终端系统的本质进行了根本性重新定义。
>
> **旧定义**（v1.0）：终端通过向零粒子量子场发送编码指令来操控零粒子，本质是"零粒子场的管理员权限"。
>
> **新定义**（v2.0→v3.0兼容）：终端**直接改写量子场的基础参数与状态**，操作层面比零粒子類和神粒子類都更底层。零粒子類只是改变局部量子场的一种方式——而终端绕过了所有中间层，直接触碰世界的底层代码。
>
> **v3.0魔法体系更新**：详见 `91_legacy_magic.md`。终端的作用原理与神粒子類一致（量子场参数编辑），但终端数据库中**不包含**任何关于零粒子類的信息。零和白羽均无法感知或使用常规魔法（魔力检测=F级/0）。

## 一、系统本质（量子层面的世界改写器）

### 1.1 终端是什么

- **名称**：Terminal v2.0
- **界面**：纯CLI命令行界面·显示在零的意识视野中·只有零能看到
- **语言**：英语——零的前世英文能力以"肌肉记忆"形式保留，所以他能理解含义并感觉自己"会写"
- **AI助手**：Eve ——机械式·功能导向·无感情·只返回必要信息
- **用户ID**：KAMISHIRO_REI = Observer #7
- **激活时间**：零3岁时首次接触（零独自在家发烧→终端自动启动→Y/N执行修复→退烧）
- **运行时长**：13年（到16岁入学时）

**终端的本质定义**：

终端不是魔法道具，不是零粒子類操控装置，也不是神粒子類共鸣器。终端是一个**量子场参数编辑器**——它允许持有者直接读取、修改、并持续控制周围量子场的基础状态。

类比来说：
- 这个世界的普通物理法则 = 操作系统的"默认设置"
- 零粒子類的存在 = 在操作系统上安装了一个"插件"，这个插件可以局部改变设置（通过第五力/粒子相互作用）
- 神粒子類 = 一个更强大的"管理员插件"（直接修改参数）
- **终端 = 直接编辑操作系统的源代码 / 内核参数**

这就是为什么理论上终端比任何已知的零粒子類或神粒子類手段都强大——因为它不依赖任何中间层。

**⚠️ v3.0关键约束**：终端的数据库中**没有**关于零粒子類的任何信息。终端只知道地球的标准模型物理学。这意味着终端无法自动识别这个世界的魔法物质或现象——零必须通过观察和"喂书"来让终端学习。详见 `91_legacy_magic.md` 第11节。

### 1.2 量子操控 vs 零粒子類 vs 神粒子類

| 操控方式 | 作用层面 | 原理 | 上限 |
|----------|---------|------|------|
| **常规魔法（零粒子類）** | 局部量子场 | 通过魔法物质（第五力）或体内MCCs代谢→改变周围量子场状态 | 受限于魔法等级考证和道具品质 |
| **神粒子類操控** | 更大范围量子场 | 神粒子類直接修改量子场参数→可影响更大范围 | 受限于载体稳定性（目前仅小雪可用） |
| **终端（Terminal v2.0）** | **量子场基础参数本身** | 直接读写量子场的状态变量·不经过任何中间媒介 | **理论上仅受采样精度和计算能力限制** |

**层级关系图示**：

```
┌─────────────────────────────────────┐
│         现实世界（现象层）            │
│  ← 我们看到的"魔法效果"都在这里表现   │
├─────────────────────────────────────┤
│       零粒子類 / 神粒子類 层          │
│  ← 常规魔法和神之力通过这一层操作      │
│  ← 需要魔法道具 / MCCs代谢 / 血统     │
├─────────────────────────────────────┤
│         量子场基础层                  │
│  ← 终端直接操作的层面                 │
│  ← 不需要任何中间钥匙·直接读写        │
│  ← 这就是为什么F级的零也能使用         │
└─────────────────────────────────────┘
```

**关键推论**：

> **主角理论上各种能力都能用，存在的或不存在的都能用。只要采样精度足够、代码正确、且没有观测者干扰，零可以通过终端执行任何类型的量子场改写操作。**

### 1.3 为什么需要采样

这是整个终端系统最关键的**瓶颈所在**。

#### 连续 vs 离散的鸿沟

这个世界的物理信息——尤其是量子场信息——是**连续变化的模拟信号**（Analog Signal）。温度在连续变化、声音在连续波动、物质内部的量子态在连续演化、空间曲率在连续扭曲……一切都是模拟的、连续的、高维度的。

但终端是一个**数字计算设备**。它只能处理离散的数字数据（Digital Data）。

要把连续的模拟信号转化为离散的数字数据，就需要一个过程：**采样（Sampling）**。

#### 采样的核心难点

- **量子场信息的采样极其困难**：量子场的信息维度极高（涉及位置、动量、自旋、纠缠态等多个自由度），且遵循不确定性原理
- **采样精度决定一切**：如果采样不够精确，后续编写的代码就会基于错误的数据运行，结果可能完全偏离预期——甚至产生危险的副作用
- **实时性要求**：量子场的状态在不断变化，采样数据有时效性，过时的数据会导致程序失效

#### 为什么这成了最大的瓶颈

终端虽然理论上有改写量子场基础参数的能力，但**它无法凭空知道当前量子场的状态是什么**。就像你有一个超级强大的文本编辑器可以修改任何文件的任何内容——但你首先需要打开文件、读取内容，才能知道该改哪里、改成什么。

**采样 = 打开文件并读取内容。**

没有采样，终端就是一个对着空白屏幕敲键盘的超级计算机——能力再强也无从发力。

### 1.4 亚空间与预设基础采样器

#### 亚空间（Subspace）

终端并非存在于零的肉体或意识中某个具体位置——它存在于一个被称为**"亚空间"**的量子折叠区域里。亚空间的特性：

- 与现实量子场存在**单向信息通道**：亚空间可以感知现实量子场（用于采样），也可以向现实量子场发送控制信号（用于执行）
- **时间流速不同**：亚空间内的时间流速略快于现实（约1.05:1），这使得编译等计算操作在亚空间内完成时，现实中只过去了稍短的时间
- **隔离性**：亚空间不受外部观测者的直接影响（但前台模式下零的意识需要同时锚定两个空间，这导致了观测者效应）
- **存储能力**：亚空间内存在有限的数据存储区，用于保存预编译的程序、采样历史、Eve数据库等

#### 预设基础采样器（Built-in Samplers）

原始终端在亚空间中预设了几个**基础采样器（Basic Samplers）**。这些采样器是Phantasm计划在投放前预先编写并植入的，不需要Token即可调用，但功能相对基础：

| 采样器名称 | 功能 | 可获取数据 | 精度等级 | Token消耗 |
|-----------|------|-----------|---------|----------|
| `sampler_temp` | 温度采样器 | 环境温度·目标物体表面温度·体温 | ★★★☆☆ | 0 |
| `sampler_audio` | 声音采样器 | 声波频率·振幅·声源方向定位 | ★★★☆☆ | 0 |
| `sampler_position` | 位置采样器 | 自身三维坐标·相对位移·速度矢量 | ★★★★☆ | 0 |
| `sampler_phys` | 基础物理量采样器 | 重力加速度·大气压·湿度·电磁场强度 | ★★☆☆☆ | 0 |
| `sampler_bio` | 生物体征采样器（自身） | 心率·体温·体表电活动·基础代谢指标 | ★★★★☆ | 0 |
| `sampler_nullfield` | 零粒子類场密度采样器 | 局部CaP（载能粒子）浓度·第五力波动·方向性 | ★★☆☆☆ | 0 |

**关于 sampler_nullfield 的 v3.0 说明**：
- 此采样器检测的是**自由态载能粒子（CaP）的密度和波动**
- **但零无法直接"感知"这些数据**——采样器输出的是纯数值，零需要自行解读
- 对零而言，这些数字就像"阅读一份看不懂的语言的天气预报"——他知道数据在那里，但无法像原生魔法使用者那样直观地"感受"到
- 白羽的终端可以通过传感器数据**间接推断**CaP活动（温度异常、电磁异常等），但这不是"感知"
- 详见 `91_legacy_magic.md` 第5节（隐形魔法场）和第11.6节（两人都不能感知魔法）

**关于精度的说明**：
- ★★★★★ = 接近量子极限精度（目前无法达到）
- ★★★★☆ = 足够大多数精细操作使用
- ★★★☆☆ = 适合一般用途，复杂操作可能产生偏差
- ★★☆☆☆ = 仅适合粗略判断，精细操作不可靠
- ★☆☆☆☆ = 参考级精度，仅能感知"有/无"

**基础采样器的局限性**：

这些预设采样器能够满足日常基本需求（如感知温度变化、定位自身位置、监测身体状况），但对于复杂的终端操作（如精确操控某块区域的量子场结构、分析某种未知物质的内部组成）来说，精度远远不够。

要获得更高精度的采样能力，必须**自己设计和制作新的采样器**——而这需要消耗Token来购买Eve的辅助设计服务。

## 二、三步使用流程

终端的每一次有效使用都必须严格遵循以下三步流程。跳过任何一步都会导致操作失败或产生不可预测的结果。

```
┌──────────┐    ┌──────────────┐    ┌───────────────┐
│ 第一步    │ → │ 第二步        │ → │ 第三步         │
│  采样     │    │ 编码与编译    │    │ 自动化运行     │
│ Sampling │    │ Code&Compile │    │ Automation    │
│          │    │              │    │               │
│ 消耗:体力 │    │ 消耗:Token   │    │ 消耗:体力      │
│ (无Token) │    │ (+可选Eve)   │    │ (无Token)      │
└──────────┘    └──────────────┘    └───────────────┘
```

### 2.1 第一步：采样（Sampling）

#### 采样原理

当零开启终端后，第一步永远是**调用采样器感知周围的量子场**。采样器像探针一样伸入现实量子场中，捕获特定类型的信息，然后将连续的模拟信号转换为离散的数字数据，最终显示在终端界面上。

**采样过程的详细步骤**：

1. **零在终端CLI中输入采样命令**（例如 `$ sampler_temp --target radius:5m --interval:100ms`）
2. **终端向亚空间发送采样请求**
3. **对应采样器激活**，其"探针"延伸至指定范围的现实量子场
4. **探针捕获连续模拟信号**（如温度场的连续分布）
5. **模数转换（A/D Conversion）**：将连续信号按设定的采样率和量化精度转化为离散数字
6. **数据处理**：整理为结构化数据（数组、矩阵、张量等形式）
7. **输出至终端界面**：以数字/图表形式呈现给零

#### 基础采样器列表及详细用法

##### `sampler_temp` — 温度采样器

```bash
# 语法
$ sampler_temp [选项]

# 常用选项
--target <范围>        # 采样目标区域（radius:5m = 半径5米球体 / object:hand = 目标物体）
--interval <毫秒>      # 采样间隔（默认1000ms）
--precision <等级>     # 精度模式（low/medium/high，越高消耗体力越大）
--output <格式>        # 输出格式（raw数值 / heatmap热力图 / trend趋势线）
--duration <秒>        # 持续采样时间（默认一次性采样）

# 示例
$ sampler_temp --target radius:10m --interval:500ms --output heatmap
# → 以500ms间隔持续采样半径10米范围内的温度分布，输出热力图
```

**输出示例**：
```
[SAMPLER_TEMP] Active. Target: r=10m sphere. Interval: 500ms.
[00:00.5] Center(0,0,0): 22.3°C | N(0,0,10): 21.8°C | Gradient: -0.05°C/m
[00:01.0] Center(0,0,0): 22.3°C | N(0,0,10): 21.8°C | Stable.
[00:01.5] Center(0,0,0): 22.4°C | Anomaly detected at (3.2, -1.1, 0.5): +2.7°C Δ
^C
[SAMPLER_TEMP] Stopped. Data saved to ./samples/temp_XXXX.log
```

##### `sampler_audio` — 声音采样器

```bash
$ sampler_audio [选项]

--directional          # 启用方向定位（额外消耗体力）
--freq_range <Hz-Hz>   # 频率范围过滤（如 20-20000 为全频段）
--threshold <dB>       # 触发阈值（低于此值的不记录）
--record               # 录制原始音频数据（占用存储）
```

##### `sampler_position` — 位置采样器

```bash
$ sampler_position [选项]

--reference <坐标系>   # 参考坐标系（absolute绝对 / relative相对某点）
--velocity             # 同时输出速度矢量（微分计算）
--track <目标ID>       # 追踪已标记目标的运动轨迹
```

##### `sampler_phys` — 基础物理量采样器

```bash
$ sampler_phys [选项]

--all                  # 全量采集（重力+气压+湿度+电磁场）
--gravity              # 仅重力加速度
--pressure             # 仅大气压
--humidity             # 仅相对湿度
--emf                  # 仅电磁场强度
```

##### `sampler_bio` — 生物体征采样器（自身）

```bash
$ sampler_bio [选项]

--hr                   # 心率
--temp                 # 核心体温
--eda                  # 皮肤电活动（可反映压力/紧张程度）
--metabolic            # 基础代谢率估算
--full                 # 全套生物体征面板
```

##### `sampler_nullfield` — 零粒子類场密度采样器（v3.0更新）

```bash
$ sampler_nullfield [选项]

--density              # CaP（载能粒子）浓度（单位：mP）
--frequency            # 第五力主导波动频率（对应属性倾向推测）
--direction            # 场的方向性/流动趋势
--anomaly              # 异常检测模式（自动标记异常波动点）
```

**v3.0重要说明**：此采样器可以检测到CaP场的**数值数据**，但：
- **零无法像原生魔法使用者那样"直观感受"**这些数据的意义
- 数值对零而言是抽象的——他需要通过经验学习什么数值范围对应什么现象
- 这就像一个色盲患者可以使用光谱仪读取波长数值，但无法真正"看到"颜色
- 白羽的终端有类似的限制，但她的Eve AI可以帮助解读数值模式

#### 自制采样器（Custom Samplers）

当基础采样器的精度不足以支撑复杂操作时，零需要**设计并制作全新的专用采样器**。

##### 制作流程

1. **需求分析**：明确需要采集什么类型的信息、需要多高的精度、采样范围多大
2. **Eve辅助设计**（消耗Token）：向Eve提交采样器设计请求，Eve基于量子场理论和现有采样器架构生成设计方案
3. **方案评审**：零审核Eve生成的方案——由于前世博士级的知识储备，他能判断方案的可行性
4. **原型构建**：在亚空间中编译采样器原型代码
5. **实验测试**（消耗Token）：在实际环境中测试采样器的性能——精度是否达标？数据是否有噪声？边界情况如何？
6. **迭代优化**：根据测试结果反复调整参数和算法
7. **注册入库**：测试通过的采样器被保存到终端的采样器库中，之后可直接调用

##### Token消耗标准

| 制作阶段 | Token消耗 | 说明 |
|----------|-----------|------|
| Eve初步设计 | 200-500w | 生成基础架构方案 |
| 详细设计+算法优化 | 500-1000w | 复杂采样器的高级设计 |
| 模拟测试 | 300-500w | 在亚空间隔离环境中验证 |
| 实地校准 | 200-500w | 在真实环境中的精度校准 |
| **总计** | **500-2000w** | 取决于采样器复杂度 |

##### 已知自制采样器（剧情进展中逐步解锁）

| 采样器名称 | 用途 | 精度 | 制作时期 | Token消耗 |
|-----------|------|------|---------|----------|
| `sampler_qft_detail` | 量子场精细结构采样 | ★★★★☆ | 废坑实验期（~12岁） | ~1200w |
| `sampler_material_comp` | 物质内部量子结构成分分析 | ★★★☆☆ | 入学前（~15岁） | ~800w |
| `sampler_spatial_curvature` | 空间曲率/拓扑异常检测 | ★★★☆☆ | 特别考试准备期 | ~1500w |

#### 体力消耗机制

**采样过程不消耗Token，纯粹消耗零的个人能量——主要表现为身体热量（体温）的下降。**

| 采样行为 | 体力消耗程度 | 体感表现 |
|----------|-------------|---------|
| 单次快速采样（<1秒） | 极微弱 | 几乎无感 |
| 持续低精度采样（数分钟） | 轻微 | 手指尖微凉 |
| 持续高精度采样（数分钟） | 中等 | 明显感到身体发冷 |
| 多采样器同时运行 | 较强 | 手脚冰凉·轻微颤抖 |
| 高精度+大范围+长时间 | 强烈 | 体温明显下降·嘴唇发白·思维迟缓 |

**体力恢复**：休息+进食+睡眠。正常饮食条件下约30分钟可恢复一次中等强度的采样消耗。连续过度采样而不休息会导致：

1. 体温过低（Hypothermia）
2. 意识模糊
3. 手抖导致输入错误
4. 强制终止所有正在运行的采样器和程序（安全机制）
5. 昏迷（极端情况）

### 2.2 第二步：编码与编译（Code & Compile）

#### 为什么写代码这么难

采样完成后，零获得了描述当前量子场状态的数字化数据。下一步是**编写代码来描述想要实现的量子场变化**——这一步的难度超乎想象：

##### 数据的高维复杂性

采样输出的数据不是简单的数字列表——它是**高维张量数据**。一个简单的"改变房间温度"的操作，背后涉及的可能是：
- 三维空间中每个点的温度值（3D标量场）
- 温度随时间的演化规律（时间维度）
- 热传导方程的边界条件
- 空气密度变化导致的对流效应
- 周围物质的比热容和热导率差异
- 零粒子场对热力学行为的修正项

**这些数据需要在纯文本CLI中手动处理和分析。**

##### 纯CLI环境的困境

零的终端界面是**纯黑底白字的命令行**（Bash风格）。这意味着：

- **没有图形化界面（GUI）**：没有按钮、没有菜单、没有可视化编辑器
- **没有集成开发环境（IDE）**：没有代码补全（初期）、没有语法高亮、没有调试器、没有断点
- **没有鼠标操作**：所有操作通过键盘输入命令完成
- **屏幕尺寸有限**：意识视野中的终端窗口大约相当于80列×24行的文本终端
- **没有复制粘贴**（初期）：每行代码必须手打

**类比**：让你用vi编辑器在一个纯黑的远程终端上——通过SSH连接到一个没有互联网的服务器上——手写一个完整的量子物理模拟程序。而且你只能看到屏幕上那几十行文字。

##### 主角的优势与局限

**优势**：
- 前世博士级的数学和物理学知识——理解量子场论、控制论、信息论
- CLI操作是前世的核心技能之一——肌肉记忆级别的熟练度
- 13年的终端使用经验——积累了大量可复用的代码模板和脚本库
- 天才级的逻辑思维能力——能在脑中进行复杂的数学推演

**局限**：
- 记忆不完整——前世知识以碎片形式保留，需要"触发"才能回忆
- 人脑的短期记忆有限——无法同时处理过于庞大的代码量
- 手写速度受限于意识中的"虚拟键盘"输入速率
- 没有团队协作——一个人承担全部开发工作
- 时间压力——战斗等紧急场景下不可能从容编程

#### Token购买Eve辅助

面对上述困境，**烧Token来获得Eve的AI辅助**是最关键的效率提升手段。

##### Eve在编码阶段能做什么

| 辅助功能 | 说明 | 适用场景 |
|----------|------|---------|
| **代码补全（Autocomplete）** | 根据上下文预测并补全下一行/下一个函数 | 所有编码场景 |
| **语法检查（Syntax Check）** | 实时检查代码语法错误 | 编码过程中 |
| **模板生成（Template Gen）** | 根据需求自动生成代码框架/模板 | 新建操作脚本时 |
| **调试建议（Debug Suggest）** | 分析报错信息并提供修复建议 | 出错时 |
| **算法优化（Algorithm Opt）** | 分析代码效率并提出优化方案 | 性能敏感场景 |
| **文档查询（Doc Lookup）** | 查询终端内置API文档和用法说明 | 不确定API时 |
| **代码重构（Refactor）** | 自动优化已有代码的结构和可读性 | 维护旧脚本时 |
| **模拟运行（Dry Run）** | 在隔离环境中预演代码执行效果 | 关键操作前的验证 |

##### 不烧Token也能写吗？

**技术上可以，但实际上几乎不可行。**

- 最简单的操作（如移动一个小物体）可能只需要5-10行代码，有经验的零可以在1-2分钟内手写完成
- 中等复杂度的操作（如给自己施加一个物理强化buff）可能需要50-200行代码，手写耗时10-30分钟且容易出错
- 复杂操作（如精确操控某区域的量子场结构实现隐身效果）可能需要500-2000+行代码，**不靠Eve几乎不可能在合理时间内完成**
- 超高复杂度操作（如空间折叠、大规模量子重构）可能需要数千甚至上万行代码——即使有Eve辅助也需要数小时

##### Token消耗标准（编码阶段）

| 操作类型 | Token消耗 | 典型场景 |
|----------|-----------|---------|
| 手写简单代码（无Eve） | 0 | 移动小物体·开关灯·简单温度调节 |
| Eve辅助编码（简单操作） | 20-50w | 代码补全+语法检查+简单调试 |
| Eve辅助编码（中等操作） | 100-300w | 模板生成+算法选择+中等规模调试 |
| Eve辅助编码（复杂操作） | 500-1000w | 完整算法设计+深度优化+多轮迭代调试 |
| Eve辅助编码（超高复杂度） | 1000-2000w+ | 从零开始构建全新操作逻辑+大量模拟验证 |

#### 编译流程

代码编写完成后，必须经过**编译（Compile）**才能转化为可执行的量子场操作序列。

##### 编译的作用

将零用类C/类Python风格的终端脚本语言编写的高级代码，翻译为量子场可以直接"理解"的低级操作指令序列。这个过程包括：

1. **词法分析与语法解析**：检查代码结构的合法性
2. **语义分析**：验证代码逻辑的一致性和完整性
3. **量子场映射**：将抽象的操作意图映射为具体的量子场参数修改指令
4. **优化**：简化指令序列、消除冗余操作、优化执行路径
5. **安全性检查**：检测可能导致危险副作用的代码模式
6. **生成可执行二进制**：输出可在亚空间虚拟机中运行的编译产物

##### 编译的Token消耗

| 编译类型 | Token消耗 | 说明 |
|----------|-----------|------|
| 基础编译（无优化） | 10-20w | 快速编译，不做深度优化 |
| 标准编译（含Eve优化） | 50-200w | Eve进行通用优化 |
| 深度优化编译 | 200-500w | 针对特定场景的极致优化（战斗脚本常用） |
| 安全审计编译 | 100-300w | 额外的安全性验证（高风险操作推荐） |

编译完成的程序会被保存在亚空间的存储区中，分为以下几类：

```
│   ├── barrier        # 防御屏障
│   ├── mana_blade     # 零粒子刃（废弃命名，实际为量子场凝聚刃）

│   └── counter        # 反击响应
```

### 2.3 第三步：自动化运行（Automation / Execution）

#### 控制论原理

编译后的程序在第三步被执行时，运用的核心原理来自**控制论（Cybernetics）**：

- 程序作为一个**控制器（Controller）**持续向量子场发送修正信号
- 量子场的实际状态不断被反馈回控制器（通过采样器）
- 控制器根据目标状态与实际状态的差距，动态调整输出信号
- 整个系统形成一个**闭环反馈回路（Closed-loop Feedback）**

这种机制使得终端的效果看起来像是"魔法"——实际上它是**量子场的受控偏移（Controlled Deviation）**。

```
        ┌──────────────────┐
        │   目标状态 Target │
        │  （代码中定义的   │
        │   期望量子场状态） │
        └────────┬─────────┘
                 │ 差距
                 ▼
        ┌──────────────────┐
        │   控制器 Controller│
        │  （编译后的程序）  │
        └────────┬─────────┘
                 │ 控制信号
                 ▼
    ┌────────────────────────┐
    │   量子场 Quantum Field  │◄─── 现实世界
    │  （被控制的物理对象）    │
    └────────┬───────────────┘
             │ 反馈信号（通过采样器）
             ▼
    ┌──────────────────┐
    │   传感器 Sensor   │
    │  （采样器模块）    │
    └──────────────────┘
```

#### 体力消耗（运行阶段）

**程序的运行同样不消耗Token，纯粹消耗零的身体热量。**

运行阶段的体力消耗取决于以下几个因素：

| 因素 | 低消耗 | 高消耗 |
|------|--------|--------|
| 程序强度 | 微调类（小幅参数修改） | 强干预类（大幅偏离自然状态） |
| 作用范围 | 小范围（<1米半径） | 大范围（>10米半径） |
| 持续时间 | 短暂（<10秒） | 长时间（>10分钟） |
| 精细度 | 粗略（容许较大误差） | 精确（要求极小误差） |
| 并发数 | 单个程序运行 | 多个程序同时运行 |

**典型场景的体力消耗参考**：

| 运行场景 | 消耗等级 | 体感 | 持续时间上限 |
|----------|---------|------|-------------|
| 维持mana_veil伪装 | 极低 | 几乎无感 | 理论上无限（需定期补充热量/食物） |
| 小范围温度微调（±2°C） | 低 | 轻微发凉 | ~2小时 |
| 个人物理强化buff | 中等 | 明显发冷·手脚冰凉 | ~30分钟 |
| 中等范围隐身效果 | 中高 | 嘴唇发白·轻微颤抖 | ~15分钟 |
| 量子场凝聚武器（mana_blade） | 高 | 强烈寒冷感·体力快速流失 | ~5分钟（战斗中） |
| 大范围结界/屏障 | 很高 | 体温急剧下降·视野模糊 | ~3分钟 |
| 空间级别操作 | 极高 | 接近生理极限·随时可能昏迷 | ~30秒-1分钟 |

#### 持续性 vs 瞬时性程序

根据执行特性的不同，终端程序可分为两大类：

##### 持续性程序（Persistent Programs）

- **特点**：需要持续运行才能维持效果，一旦停止则效果消失
- **运行方式**：采样器持续采集数据 → 控制器持续输出修正信号 → 量子场维持在目标状态
- **典型例子**：
  - `mana_veil`（常驻伪装脚本）：持续模拟D级零粒子痕迹
  - `stealth_field`（隐身效果）：持续修改光线在特定区域的传播路径
  - `phys_boost`（物理强化buff）：持续增强肌肉力量和反应速度
  - `barrier`（屏障/结界）：持续维持一道防御性的量子场壁垒
  - `temp_control`（温度控制）：持续维持某区域的温度在设定值
- **可以后台运行**（详见第三节）
- **总消耗 = 单位时间消耗 × 运行时间**

##### 瞬时性程序（Instantaneous Programs）

- **特点**：一次性执行完毕，释放出瞬间的效果后结束
- **运行方式**：在极短时间内集中输出大量控制信号，使量子场发生剧烈但短暂的变化
- **典型例子**：
  - `quantum_burst`（量子冲击）：瞬间释放高能量冲击波
  - `spatial_fold`（空间折叠）：瞬时压缩两点间的空间距离
  - `field_collapse`（场坍缩）：使局部量子场瞬间坍缩产生吸力/斥力
  - `matter_reconstruct`（物质重构）：短时间内重组物质的分子结构
- **不能后台运行**——必须在前台模式下执行
- **总消耗集中在极短的时间内释放**——对身体的瞬时负担极大

## 三、前台模式与后台运行

### 3.1 前台模式（Foreground Mode）—— 终端可见

#### 什么是前台模式

当零**主动开启终端的CLI界面**使其显示在意识视野中时，即为前台模式。此时：

- 零可以直接看到终端的输出信息
- 可以实时输入命令
- 可以进行交互式操作（如调试、单步执行）
- **受到观测者效应的影响**
- **可以使用完整的终端功能**（包括瞬时高能量程序）

#### 前台模式的观测者效应

| 周围观测者人数 | 终端稳定性 | 有效输出能力 | 相当于魔法等级 | 典型场景 |
|---------------|-----------|------------|--------------|---------|
| **0人** | **100%** | **理论上限** | **S级~超S级** | 独处·废坑深处·深夜自室 |
| **1人** | **~90%** | **高** | **A+~S级** | 小雪在隔壁房间·仅有亲近之人 |
| **2-3人** | **~82%** | **较高** | **A~A+级** | 小群体私密对话 |
| **5人** | **~71%** | **中等** | **B+~A级** | 少人数训练·小组讨论 |
| **10人** | **~58%** | **中低** | **B~B+级** | 小教室·走廊 |
| **30人** | **~41%** | **低** | **C~B级** | 教室上课·食堂 |
| **50人** | **~30%** | **很低** | **D~C级** | 大教室·礼堂后排 |
| **100人** | **~18%** | **极低** | **D~E级** | 学园集会·操场集合 |
| **200+人** | **<8%** | **近乎失效** | **F级以下** | 全校集会·竞技场比赛·公共场合 |

#### 观测者效应的深层原因

**为什么开启终端界面时会受到观测者效应？**

当零处于前台模式时，他的意识同时存在于两个层面：

1. **量子操作层**：意识的一部分深入亚空间，与终端和量子场进行交互
2. **现实感知层**：意识的另一部分仍然锚定在现实世界，感知周围环境和他人存在

这种**双重意识状态**使得零的量子签名在两个层面之间产生了"干涉条纹"——类似于光学中的双缝干涉实验。周围其他人的意识（尤其是具有魔法能力的人——即体内MCCs数量较多者）会对这种不稳定的状态产生**量子干涉**，表现为：

- 终端的输出信号中混入"噪声"
- 采样器的精度下降
- 编译后的程序执行效率降低
- 严重时程序直接崩溃或产生不可预期的结果

**观测者数量越多 → 干涉信号的来源越多 → 噪声叠加越强 → 终端性能衰减越严重**

#### 前台模式下的战术含义

- **公开场合零必须"扮猪"（F级）**：200人以上的场合终端近乎失效，零表现得像个普通人反而最安全
- **私下场合零可以"吃老虎"（S级）**：独处时终端达到100%性能，可以施展远超外表能力的操作
- **团队战中利用队友分散注意力**：让队友吸引观测者的视线，减少对零的直接观测
- **利用盲区和死角**：在人群中找到视线死角进行隐蔽操作
- **提前预编译**：在人少的时候把复杂的程序编译好，人多的时候只需执行（执行阶段受观测者影响较小）

### 3.2 后台模式（Background Mode）—— 终端不可见

#### 什么是后台模式

已经编译好的持续性程序可以在**不开启终端界面**的情况下自动运行。此时：

- 终端的CLI界面**不显示**在零的意识视野中
- 零不需要主动关注程序的运行状态
- 程序在亚空间中自动执行
- **不受观测者效应影响**（因为零的意识不需要同时锚定两个空间）
- 但有严格的**程序类型限制**

#### 后台模式的运行原理

后台运行时，零的意识**完全沉浸在量子操作层**中——程序由亚空间内的虚拟机自动驱动，采样器自动采集数据，控制器自动输出修正信号。整个过程不需要零的显式参与（类似于操作系统的后台守护进程/deamon）。

正因为零的意识不需要同时维持"现实感知层"和"量子操作层"的双重状态，所以**不存在双重意识带来的量子干涉**——观测者效应因此不适用。

#### 后台模式的限制条件

**不是所有程序都可以后台运行。** 只有满足以下条件的程序才被允许进入后台模式：

| 允许后台运行的条件 | 说明 |
|-------------------|------|
| **必须是持续性程序** | 瞬时性程序不能后台运行 |
| **强度必须在阈值以下** | 不能超过"低强度"标准（由终端自动判定） |
| **必须有完善的异常终止机制** | 程序必须包含安全退出逻辑 |
| **必须经过至少一次前台成功运行验证** | 未经验证的程序禁止后台运行 |
| **采样器必须是已注册的稳定版本** | 实验性采样器不允许在后台使用 |

##### 可以后台运行的程序示例

- ✅ `mana_veil`（常驻伪装）：极低强度·长期运行·完善的异常处理
- ✅ `phys_boost_light`（轻度物理强化）：低强度buff·可持续
- ✅ `temp_maintain`（温度维持）：小范围·低幅度·渐进式
- ✅ `monitor_passive`（被动监控）：仅采集数据·不主动干涉
- ✅ `barrier_mini`（小型屏障）：防御性·低强度·持续性

##### 不能后台运行的程序示例

- ❌ `quantum_burst`（量子冲击）：瞬时高能量·超出强度阈值
- ❌ `spatial_fold`（空间折叠）：空间级别操作·风险过高
- ❌ `matter_reconstruct`（大规模物质重构）：计算量和能量输出过大
- ❌ `combat_full_combo`（完整战斗连招）：包含瞬时高能量子程序
- ❌ 任何未经前台验证的新程序

```bash
$ bgstop <pid_or_name>

# 强制停止（紧急情况）
$ bgstop --force <pid_or_name>

# 停止所有后台程序
$ bgstop --all
```

#### 多任务并发

多个后台程序可以同时运行，但有**并发上限**：

| 并发数量 | 限制说明 |
|---------|---------|
| 1-2个 | 无限制·正常运行 |
| 3个 | 开始感受到明显的叠加热量消耗 |
| 4个 | 体力消耗加速·需要频繁补充热量 |
| 5个 | **最大并发数**·接近安全上限 |
| >5个 | 终端拒绝启动新程序·提示"RESOURCE LIMIT REACHED" |

**热量消耗叠加规则**：多个后台程序同时运行时，总体力消耗约为各程序单独消耗之和的**1.2-1.5倍**（因为程序间存在资源竞争和协调开销）。

### 3.4 体力管理与过载风险

#### 体力监控系统

终端内置了基础的体力监控系统（基于`sampler_bio`的持续数据）：

#### 过载警告等级

| 等级 | 体温降幅 | 系统反应 | 零的体感 |
|------|---------|---------|---------|
| NORMAL（正常） | Δ<0.5°C | 无特殊提示 | 无不适 |
| CAUTION（注意） | Δ0.5-1.0°C | 黄色警告 | 轻微发凉 |
| WARNING（警告） | Δ1.0-1.5°C | 橙色警告·建议减少负载 | 明显发冷·手指微僵 |
| DANGER（危险） | Δ1.5-2.0°C | 红色警告·强制建议停止 | 嘴唇发白·手抖·思维变慢 |
| CRITICAL（临界） | Δ>2.0°C | **强制终止所有程序** | 视野模糊·剧烈颤抖·接近昏迷 |

#### 强制终止机制

当体力降至CRITICAL等级时，终端会**无条件强制终止所有正在运行的前台和后台程序**——无论零是否同意。这是Phantasm计划植入的安全保护机制，目的是防止零因过度使用终端而导致永久性身体损伤。

强制终止后的恢复期：
- 立即效果：所有程序停止·体力消耗归零
- 短期（1-2小时）：极度疲惫·嗜睡·无法再次启动终端
- 中期（半天）：虚弱·食欲大增·需要大量进食
- 长期（1-2天）：完全恢复·无永久损伤（除非多次触达临界）

| 对比维度 | 魔法使用者（原住民） | 终端使用者（零） |

| **便携性** | **极佳**（身体本身就是媒介·无需额外工具） | **受限**（需要意识进入终端·需要一定的精神集中） |
| **力量上限** | 受限于血统/亲和性等级（天花板明确） | **理论上极高**（仅受采样精度和计算能力限制·天花板不明） |
| **学习成本** | 从小训练即可·直觉导向·熟能生巧 | 需要**量子物理+编程+控制论**的跨学科知识·门槛极高 |

### 4.2 零的核心弱势和核心优势

#### 核心弱势：便利性差距

**终端不如直接使用魔法方便——这是零在整个故事中最根本的短板之一。**

具体表现在：

1. **准备时间长**：从意识到需要使用能力到实际产生效果之间，存在数十秒到数分钟的"真空期"。在这个真空期内，一个普通的C级魔法使已经完成了三次攻击。
2. **无法应对突发状况**：遭遇偷袭、 ambush、意外事件时，来不及走完完整的采样-编码-编译-运行流程。
3. **观测者硬限制**：有人看着就做不了——这从根本上限制了零在公开场合的能力发挥。
4. **体力硬限制**：用久了身体扛不住——不像魔法使那样可以通过冥想来恢复零粒子，零的体力恢复需要真实的休息和进食。
5. **学习曲线陡峭**：每一个新效果的实现都需要从头研究量子场的特性、设计采样器、编写代码、测试调试——时间成本巨大。

#### 核心优势：理论上限

**但终端的理论上限远高于任何已知的魔法体系——这也是零的核心优势所在。**

具体表现在：

1. **全能性潜力**：理论上只要能写出代码，就能实现任何效果。不受学派限制、不受属性限制、不受血统限制。
2. **精细操作能力**：可以达到魔法使无法企及的精度——比如精确控制某个分子的运动轨迹、精确修改某束光线的折射角度。
3. **可复现性与可优化性**：一旦某个效果的代码写好了，就可以无限次复用，并且可以不断优化改进。魔法使的能力提升往往依赖于"感觉"和"领悟"，难以系统性地优化。
4. **组合灵活性**：不同效果的代码可以组合、嵌套、并行运行——创造出这个世界从未见过的复合效果。
5. **不被常规体系认知**：终端的力量模式与这个世界的所有已知魔法体系完全不同——意味着现有的对抗手段（针对零粒子魔法的防御、侦测、封锁）对终端可能无效。

### 4.3 成长方向

整个故事中零的成长方向之一就是**提高终端使用的效率**，缩小与魔法使用者在便利性上的差距：

| 成长方向 | 具体措施 | 预期效果 |
|----------|---------|---------|
| **代码库积累** | 将常用操作封装为可复用的脚本/模板 | 减少重复编码时间·从"每次重写"变为"调用模板" |
| **采样器升级** | 制作更高精度/更快速度/更低功耗的自制采样器 | 提升数据质量·减少后续代码的复杂度 |
| **预编译策略** | 提前编译好常见场景的战斗/应用脚本包 | 战斗时只需调用·省去现场编码时间 |
| **Eve能力提升** | 通过累计Token使用提升权限等级·解锁更多Eve功能 | 获得更强的AI辅助·进一步缩短编码时间 |
| **简易GUI开发**（后期） | 自己编写一个基本的图形化前端 | 降低CLI操作的门槛·提高输入效率 |
| **后台程序优化** | 将更多操作转化为可后台运行的持续性程序 | 减少前台模式的使用频率·规避观测者效应 |

| **注入系统解锁**（后期） | 获得直接使用魔法的能力 | 终端负责精细操作·魔法负责快速响应·两者互补 |

## 五、"注入"系统（后期伏笔）— v3.0更新版

> ⚠️ **以下内容为后期剧情伏笔·前期不可用。**
> **v3.0变更**：注入原理从"零粒子共鸣物质转移"更新为"MCC/MRP生物魔法框架下的酶/受体转移"。详见 `91_legacy_magic.md` 第6节。

### 5.1 注入原理（v3.0）

零是地球人，基因中没有能表达**魔导细胞（MCCs）**的特殊序列——这就是他F级体质的根本原因（魔力检测=0）。但后期剧情中，零可以通过一种被称为**"注入"（Injection）**的方式临时获得魔法能力。

**注入的本质**：通过**体液交流（Bodily Fluid Exchange）**，让拥有魔法能力的人体内的**魔导细胞（MCCs）相关生物活性物质**暂时转移到零的体内，使零的细胞获得临时的魔法物质代谢能力。

### 5.2 科学机制（v3.0双语言）

#### 科学语言解释

魔法能力者的体内存在完整的「基因→MCCs→MRPs→酶系统」通路（详见 `91_legacy_magic.md` 第6.1节）。零作为地球人：

1. **缺乏特殊基因序列** → 无法表达MCCs
2. **没有MCCs** → 没有MRPs（魔素受体）
3. **没有MRPs** → 无法结合/代谢CaP或魔法物质
4. **结果** → 魔力检测=F级/0

但当零与魔法能力者发生深度体液交换时：

- 对方体内的**游离MRPs（可溶性受体蛋白）**和**关键酶分子**可以通过体液转移
- 这些物质可以**临时附着在零的普通细胞表面**，赋予其临时的CaP结合能力
- 同时转移的还有**魔法物质前体**（对方MCCs中储备的已代谢中间产物）
- 使零在一段时间内获得**临时的、低效的**魔法物质代谢能力

#### 世界语言等价描述

| 科学步骤 | 世界语言 | 常见说法 |
|---------|---------|---------|
| MRP/酶转移 | "魔力传承""力量分享" | "他的力量流入了我的身体" |
| 临时获得代谢能力 | "短暂觉醒""临时开窍" | "我突然能用魔法了（虽然很弱）" |
| 效果消退 | "力量散去""回归原状" | "感觉又变回普通人了" |

### 5.3 触发条件

**注入的发生需要同时满足以下条件**：

1. **深度的体液交换**：不仅仅是皮肤接触或接吻，而是需要达到**"同房"（性行为）**级别的体液交流——这是唯一能使足够量的生物活性物质完成转移的途径
2. **对方必须具有有效的魔法能力**：即对方体内的MCCs密度必须达到一定阈值（D级以上即可·等级越高效果越好·MCCs越多=可转移的MRPs和酶越多）
3. **双方需要有足够的情感连接/亲密度**：情感因素会影响生物活性物质的"接受率"——单纯的物理接触不足以触发（可能是某种神经-内分泌协同效应？世界尚未研究清楚）
4. **零的身体状态需要足够健康**：极度虚弱时无法完成注入过程

### 5.4 注入的角色对象

根据故事中的角色关系，以下角色具备成为"注入源"的可能性：

| 角色 | 魔法等级 | 类型/属性 | 注入后零可获得的能力倾向 | 备注 |
|------|---------|----------|----------------------|------|
| 朱音 | A+级 | α型（燃者）·火/热力学 | 火焰操控·热能转换·力场操作 | 最可能的首次注入对象 |
| 小雪 | C级（实际A级+神粒子類适性） | ε型（愈者）+冰系·熵变 | 冰晶操控·温度控制·治愈加速 | 特殊体质·神粒子類可能有意外效果 |
| 艾露娜 | A级宗師階 | γ型（唤风者）+混合型·风/空间 | 空间操作·风力控制·领域展开 | 精灵族血统·可能有种族特性加成 |
| 真夜 | B+级 | ζ型（感者）+暗系·信息 | 信息操作·隐蔽感知·暗影力量 | 魔族（龙族后裔）血统·可能有特殊效果 |

### 5.5 注入效果（v3.0）

#### 直接效果

- **临时获得微弱魔法代谢能力**：零的魔力检测值从F级(0)临时提升至**E~D级水平**（取决于注入量和亲密度，通常为注入源等级的30%-70%）
- **获得注入源的属性倾向**：可以使用对应类型的魔法（但熟练度远低于原主人·因为零没有真正的MCCs·只是借用了对方的酶和受体）
- **可以直接使用简单魔法**（无需终端）：注入期间的魔法使用方式类似普通低阶魔法使——**即时发动·无需采样编码流程**
- **但复杂操作仍需终端辅助**：注入获得的魔法能力仅限于简单直接的效果
- **持续时间**：数小时到数天不等——取决于转移的生物活性物质的半衰期·以及零自身的代谢清除速度

#### 副作用与限制

- **效果递减**：随着时间推移·转移到零体内的MRPs和酶逐渐失活或被清除·效果减弱
- **不可叠加**：同时只能承受一种注入效果·新的注入会覆盖旧的
- **冷却期**：一次注入结束后·需要一定时间（约1-2周）让零的身体恢复基线状态才能再次接受注入
- **身份暴露风险**：注入期间零不再检测为纯粹的F级/0·可能被精密仪器检测到异常（但这个世界检测技术粗糙·实际风险较低）
- **情感绑定**：注入过程中双方会产生强烈的情感/精神连接——这可能影响人际关系
- **生理负担**：借用的酶系统在零的非特化细胞上工作效率低下·可能导致代谢副产物积累（比正常魔法使用者更容易"魔力耗尽"）

### 5.6 注入与终端的协同（v3.0）

**注入后零可以同时使用终端和简易魔法——两者的结合才是零的"完全体"形态。**

#### 分工模式

| 能力类型 | 使用方式 | 优势 |
|----------|---------|------|
| 快速响应/瞬发操作 | **直接使用注入获得的简易魔法** | 解决了终端"启动慢"的核心弱点 |
| 精细操作/复杂控制 | **使用终端** | 发挥终端"精度高"的核心优势 |
| 持续性效果维持 | **终端后台程序** | 不受观测者效应影响·低消耗长时间运行 |
| 大规模/破坏性攻击 | **简易魔法+终端联合** | 魔法提供能量基础·终端进行精确引导 |

#### 协同示例

- **战斗场景**：用注入获得的火焰魔法进行快速近战攻击·同时用终端在后台运行物理强化buff和战术分析程序
- **潜入场景**：用终端的后台隐身程序维持隐形·同时用注入获得的感知类能力增强环境侦测
- **研究场景**：用终端进行高精度的量子场数据分析·同时用注入获得的魔法能力进行实验性的魔法扰动测试

#### "完全体"的意义

注入+终端的协同不仅仅是"1+1=2"——而是创造出一种**这个世界从未见过的力量形态**：

- 普通魔法使：只有"魔法"这一种工具（受MCCs类型限制）
- 零（前期）：只有"终端"这一种工具（理论全能·使用不便·不能公开）
- 零（后期·注入后）：同时拥有"简易魔法"（快速·直觉·弱）和"终端"（全能·精细·强）两种工具·并且可以让它们协同工作

### 8.4 Token充足性分析

以初始10000w + 每日500w的资源规模来看：

**充裕的场景**：
- 日常基础采样：免费
- 简单操作的手写代码：免费
- 偶尔使用Eve辅助编码（简单操作）：每天几次·消耗可忽略
- 维持常驻的mana_veil：编译一次消耗少量Token·之后运行免费

**紧缺的场景**：
- 制作新的高级采样器：一次性消耗500-2000w
- 复杂战斗脚本的开发：单次500-1000w+
- 深度优化编译：每次200-500w
- 特别考试前的全面准备：累计可能消耗2000-4000w

**真正限制零的不是"Token够不够"而是**：

1. **观测者效应**——有人看着就做不好（最大硬限制·Token无法解决）
2. **准备时间**——突发状况来不及走完三步流程（Token也无法加速采样和编码的人类操作时间）
3. **体力限制**——用久了身体扛不住（Token无法替代热量）
4. **隐藏需求**——做了也不能被人发现（最强限制·社会层面的）

### 战略意义

mana_veil是零"扮猪吃老虎"战略的**地基**——没有它·零在日常生活中的每一个细微异常都可能被察觉。有了它：

- 在所有人眼中·零是一个"勉强有一点点零粒子共鸣的F级生"
- 零晶灯不会在他身边出现异常闪烁（因为他的"零粒子场"看起来是正常的）
- 不会被日常的零粒子感应装置标记为"异常个体"
- **但不能骗过精密的零粒子测定仪器**（如帝国魔法协会的正式测定设备）——只能防住日常观察
